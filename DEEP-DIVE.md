# Overstory Deep Dive: Architecture, Design & Code Walkthrough

## What It Is

Overstory turns a single coding session into a coordinated team of AI agents. You open Claude Code (or another supported runtime), and your session becomes the orchestrator — spawning worker agents in isolated git worktrees via tmux, coordinating them through a custom SQLite mail system, and merging their work back with tiered conflict resolution.

There is no separate daemon. No server. No UI you need to run. Your Claude Code session IS the orchestrator. CLAUDE.md instructions, shell hooks, and the `ov` CLI provide everything needed to manage a fleet of agents from the terminal.

## The Core Problem

AI coding agents are powerful but inherently single-threaded. One agent reads code, thinks, writes code, runs tests — sequentially. For large tasks that decompose into independent subtasks (migrate 20 endpoints, implement 5 unrelated features, explore a legacy codebase), you're bottlenecked by serial execution.

The naive solution — open multiple terminals with multiple agents — creates chaos. Agents overwrite each other's files. There's no coordination protocol. Merging is manual. Nobody knows what anyone else is doing.

Overstory solves this by providing three things: **isolation** (each agent works in its own git worktree), **coordination** (agents communicate through typed protocol messages), and **integration** (a merge queue with automatic conflict resolution brings the work back together).

## Architecture

### Hierarchical Delegation

Overstory uses a military-style chain of command, not a flat peer network:

```
Orchestrator (your Claude Code session)
  └── Coordinator (persistent, long-lived, spawns leads)
        └── Lead (team lead, decomposes work, spawns workers)
              ├── Scout (read-only exploration)
              ├── Builder (implementation)
              ├── Reviewer (validation)
              └── Merger (branch integration)
```

Each level has a clear role:
- **Coordinator**: Sees the big picture. Decomposes objectives into work streams. Spawns leads. Never writes code.
- **Lead**: Owns a work stream. Writes specs, spawns scouts and builders, reviews results, reports merge-ready branches upward.
- **Scout**: Explores the codebase read-only. Produces specs that describe what to build and where.
- **Builder**: The only agent that writes code. Constrained to specific files. Runs quality gates before reporting done.
- **Reviewer**: Validates builder output read-only. Catches issues the builder's tests didn't.
- **Merger**: Integrates branches back to the canonical branch through the tiered merge queue.

The hierarchy is enforced mechanically — a builder cannot spawn sub-agents, a lead cannot exceed its configured depth, and the coordinator trusts leads to decompose autonomously rather than micromanaging.

**In the code** (`src/commands/sling.ts:337-354`): The enforcement is a single condition:

```typescript
if (parentAgent === null && capability !== "lead") {
    throw new HierarchyError(
        `Coordinator cannot spawn "${capability}" directly. Only "lead" is allowed without --parent.`
    );
}
```

This one rule enforces the entire hierarchical model. Without `--parent`, only leads can be spawned. Leads then pass `--parent lead-name` when they spawn builders, scouts, and reviewers.

### Why Git Worktrees

Every agent gets its own git worktree: a full, independent working copy of the repository on its own branch. This gives you:

- **True file isolation**: Agent A editing `auth.ts` doesn't conflict with Agent B editing `auth.ts` — they're on different branches in different directories.
- **Clean canonical root**: The orchestrator's project root stays untouched. Agent overlays never overwrite the orchestrator's own CLAUDE.md.
- **Merge-ready output**: Each worktree branch is a complete, reviewable unit. The merger handles integration.
- **Simple cleanup**: `git worktree remove` deletes everything cleanly when an agent completes.
- **Zero copy cost**: Git worktrees share the `.git` object store, so creating one is nearly instantaneous regardless of repo size.

**In the code** (`src/worktree/manager.ts`): Worktree creation is a single git command:

```typescript
const branchName = `overstory/${agentName}/${taskId}`;
// git worktree add -b branchName worktreePath baseBranch
```

### Two-Layer Agent Instructions

Agents receive instructions through a composable two-layer system:

**Layer 1 — Base Definition** (`agents/builder.md`): The reusable HOW. Workflow constraints, communication protocol, quality gates, guard rules. Written once, shared across all agents of that capability.

**Layer 2 — Dynamic Overlay** (generated per spawn): The task-specific WHAT. Task ID, file scope, spec path, branch name, pre-loaded expertise from mulch. Generated by `ov sling` and written to the worktree's `.claude/CLAUDE.md`.

This separation means the base builder instructions don't repeat for every builder you spawn, while each builder still gets its unique task context. The orchestrator passes WHAT; the base definition already has HOW.

**In the code** (`src/commands/sling.ts:698-762`): The overlay generation loads the base definition, optionally fetches mulch expertise scoped to the builder's file list, builds an `OverlayConfig` struct with all substitution variables (`{{AGENT_NAME}}`, `{{TASK_ID}}`, `{{FILE_SCOPE}}`, etc.), and writes the combined result to the worktree. If overlay generation fails, the orphaned worktree is cleaned up automatically.

## The Core Flow: Spawn to Merge

The entire system can be understood by tracing one flow: what happens when you run:

```bash
ov sling task-123 --capability builder --name auth-builder
```

### Step 1: Validation Cascade

**File**: `src/commands/sling.ts:421-656`

Before anything is created, sling runs a gauntlet of checks:

| Check | What it prevents |
|-------|-----------------|
| Name uniqueness | Two agents with the same name |
| Task lock | Two agents working the same task |
| Depth limit | Runaway hierarchy (default max: 2) |
| Hierarchy validity | Coordinator spawning non-leads directly |
| Concurrency limit | Exceeding `maxConcurrent` (default: 25) |
| Parent capacity | Lead exceeding `maxAgentsPerLead` (default: 5) |
| Run session limit | Exceeding per-run agent budget |
| Root check | Running as UID 0 (Claude Code rejects `--permission-mode bypassPermissions` as root) |
| Task existence | Querying seeds/beads tracker to verify the task is open |

Each check is extracted as a pure function for testability:

```typescript
// src/commands/sling.ts:259-265
export function checkTaskLock(
    activeSessions: ReadonlyArray<{ agentName: string; taskId: string }>,
    taskId: string,
): string | null {
    const existing = activeSessions.find((s) => s.taskId === taskId);
    return existing?.agentName ?? null;
}
```

There's also a **stagger delay** (`src/commands/sling.ts:61-76`) — a sliding window that prevents agent spawn thundering. It finds the most recent session start time across all active sessions and sleeps for the remaining delay if needed (default: 2 seconds between spawns):

```typescript
export function calculateStaggerDelay(
    staggerDelayMs: number,
    activeSessions: ReadonlyArray<{ startedAt: string }>,
    now: number = Date.now(),
): number {
    if (staggerDelayMs <= 0 || activeSessions.length === 0) return 0;
    const mostRecent = activeSessions.reduce((latest, s) =>
        new Date(s.startedAt).getTime() > new Date(latest.startedAt).getTime() ? s : latest
    );
    const elapsed = now - new Date(mostRecent.startedAt).getTime();
    const remaining = staggerDelayMs - elapsed;
    return remaining > 0 ? remaining : 0;
}
```

### Step 5: Pre-send Dispatch Mail

**File**: `src/commands/sling.ts:777-798`

This is a critical race condition fix. The dispatch message is stored in SQLite **before** creating the tmux session:

```typescript
mailClient.send({
    from: parentAgent ?? "orchestrator",
    to: name,
    subject: `Dispatch: ${taskId}`,
    body: "You have been assigned task...",
    type: "dispatch",
});
```

Why before? Because when the tmux session starts, Claude Code fires its `SessionStart` hook which runs `ov mail check --inject`. If the dispatch message doesn't exist yet, the agent boots with no instructions and sits idle forever. By pre-sending, the message is already in the database when the hook fires.

### Step 6: Spawn in tmux

**File**: `src/commands/sling.ts:843-858`, `src/worktree/tmux.ts:85-142`

```typescript

`createSession()` runs:
```bash

Then retrieves the PID via `tmux list-panes -F #{pane_pid}`. The session record is persisted to SQLite **before** sending the beacon (Step 8), so hook-triggered `updateLastActivity()` can find the entry immediately (fixing race condition `overstory-036f`).

### Step 7: Wait for TUI Ready

**File**: `src/worktree/tmux.ts:483-516`

This is a polling loop — the most interesting part of the spawn sequence:

```typescript
for (let i = 0; i < maxAttempts; i++) {
    const content = await capturePaneContent(name);  // tmux capture-pane
    const state = detectReady(content);               // runtime-specific

    if (state.phase === "dialog" && !dialogHandled) {
        await sendKeys(name, "");     // dismiss dialog with Enter
        dialogHandled = true;
        continue;
    }
    if (state.phase === "ready") return true;

    if (!await isSessionAlive(name)) return false;    // agent died
    await Bun.sleep(pollIntervalMs);                  // 500ms default
}
```

It captures the tmux pane's visible text (via `tmux capture-pane -p`), passes it to the runtime's `detectReady()` which pattern-matches for UI elements (Claude Code looks for the prompt indicator), and handles three states:
- **loading**: Keep polling
- **dialog**: Send Enter to dismiss, then continue polling
- **ready**: Return — agent is accepting input

### Step 8: Send Beacon + Retry Loop

**File**: `src/commands/sling.ts:907-953`

```

The beacon is a structured one-liner:
```

```

Then the battle-tested retry loop — Claude Code's TUI sometimes swallows the Enter keystroke during late initialization:

```typescript

for (const delay of [1_000, 2_000, 3_000, 5_000]) {
    await Bun.sleep(delay);

// Verify beacon was received (overstory-3271)

}
```

This handles real-world timing issues documented with specific issue IDs (`overstory-yhv6`, `overstory-3271`).

## Messaging: Custom SQLite Mail

**Files**: `src/mail/store.ts`, `src/mail/client.ts`, `src/mail/broadcast.ts`

Agents don't talk to each other directly. They share a SQLite database (`.overstory/mail.db` in WAL mode) and discover new messages through Claude Code hooks that fire automatically on every prompt submission and tool execution.

### The Message Lifecycle

1. **Send**: Agent calls `ov mail send --to lead-a --type worker_done --body "Done"` → message stored in SQLite with `read=0`
2. **Notify**: For high-priority or protocol messages, a file marker is written to `.overstory/pending-nudges/{agent}.json` (direct tmux `send-keys` is avoided during tool execution — it corrupts I/O and kills the process)
3. **Discover**: On the receiver's next hook fire, `ov mail check --inject` reads unread messages, formats them as text, and injects them into the agent's prompt context
4. **React**: The agent sees the message as part of its next prompt and acts on it

This is **polling, not eventing** — deliberately. It's simpler, more resilient, and avoids the I/O corruption problem that plagues direct tmux injection. The tradeoff is latency (messages surface on the next hook fire, not instantly), but in practice this is seconds, not minutes.

### Message Types

Message types encode the coordination protocol:

| Type | Flow | Purpose |
|------|------|---------|
| `dispatch` | Coordinator → Agent | Initial work assignment |
| `worker_done` | Builder → Lead | Task completion signal |
| `merge_ready` | Lead → Coordinator | Branch verified, ready to merge |
| `merged` / `merge_failed` | Merger → Coordinator | Merge outcome |
| `escalation` | Any → Parent | Unresolvable issue, escalate upward |
| `status` / `question` / `error` | Any → Parent | Human-style communication |

### Broadcast Resolution

**File**: `src/mail/broadcast.ts`

Broadcast addresses (`@all`, `@builders`, `@leads`) fan out to individual messages per recipient:

```typescript
const CAPABILITY_GROUPS: Record<string, string> = {
    "@builder": "builder",
    "@builders": "builder",
    "@scout": "scout",
    "@scouts": "scout",
    "@lead": "lead",
    "@leads": "lead",
    // ... singular and plural forms both work
};
```

`@all` resolves to all active agents except the sender. `@builders` resolves to all active agents with `capability === "builder"`. Each group address produces N individual messages — no special fan-out at the receiver.

### SQLite Schema

```sql
CREATE TABLE IF NOT EXISTS messages (
  id TEXT PRIMARY KEY,
  from_agent TEXT NOT NULL,
  to_agent TEXT NOT NULL,
  subject TEXT NOT NULL,
  body TEXT NOT NULL,
  type TEXT NOT NULL DEFAULT 'status',
  priority TEXT NOT NULL DEFAULT 'normal' CHECK(priority IN ('low','normal','high','urgent')),
  thread_id TEXT,
  payload TEXT,
  read INTEGER NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
)
```

WAL mode + `busy_timeout=5000` enables concurrent reads and writes from multiple agents without locking issues. Typical query latency: ~1-5ms.

## Tiered Merge Resolution

**File**: `src/merge/resolver.ts`

When agents complete their work, branches enter a FIFO merge queue (SQLite-backed, deterministic ordering). The merger processes each branch through progressively aggressive resolution tiers:

### Tier 1 — Clean Merge

```typescript
async function tryCleanMerge(entry, repoRoot) {
    const { exitCode } = await runGit(repoRoot, ["merge", "--no-edit", entry.branchName]);
    if (exitCode === 0) return { success: true, conflictFiles: [] };
    const conflictFiles = await getConflictedFiles(repoRoot);
    return { success: false, conflictFiles };
}
```

Try `git merge --no-edit`. Works 80-90% of the time because agents work on disjoint files. Free and deterministic.

### Tier 2 — Auto-resolve

```typescript
function resolveConflictsKeepIncoming(content: string): string | null {
    const conflictPattern = /^<{7} .+\n([\s\S]*?)^={7}\n([\s\S]*?)^>{7} .+\n?/gm;
    if (!conflictPattern.test(content)) return null;
    conflictPattern.lastIndex = 0;
    return content.replace(conflictPattern, (_match, _canonical, incoming) => incoming);
}
```

Parse conflict markers with regex, keep the incoming (agent) changes by default. Also supports `merge=union` gitattribute (used by mulch's JSONL files) where both sides are kept. Still no AI cost.

### Tier 3 — AI-resolve

```typescript
const prompt = [
    "You are a merge conflict resolver. Output ONLY the resolved file content.",
    "Rules: NO explanation, NO markdown fencing, NO conversation, NO preamble.",
    "Output the raw file content as it should appear on disk.",
    "Choose the best combination of both sides of this conflict:",
    historyContext,
    content,
].join(" ");

const runtime = getRuntime(config?.runtime?.printCommand ?? config?.runtime?.default, config);
const argv = runtime.buildPrintCommand(prompt);
const proc = Bun.spawn(argv, { cwd: repoRoot, stdout: "pipe", stderr: "pipe" });
```

Spawns `claude --print` (or equivalent for other runtimes) for each conflicted file. The output is validated:

```typescript
export function looksLikeProse(text: string): boolean {
    const prosePatterns = [
        /^(I |I'[a-z]+ |Here |Here's |The |This |Let me |Sure|Unfortunately)/i,
        /^(To resolve|Looking at|Based on|After reviewing|The conflict)/i,
        /^```/m,  // Markdown fencing — the model wrapped the code
    ];
    for (const pattern of prosePatterns) {
        if (pattern.test(trimmed)) return true;
    }
    return false;
}
```

If the output looks like conversation instead of code, the tier fails and falls through to Tier 4.

**On Claude Code Max**: `claude --print` goes through the same Max subscription as interactive sessions. All tiers are effectively free on the $200/month plan.

### Tier 4 — Re-imagine

The nuclear option. Abort the merge entirely. Fetch both the canonical and branch versions of each file. Ask Claude to reimplement the agent's changes on top of the canonical version from scratch.

```typescript
const prompt = [
    "Reimplement the changes from the branch version onto the canonical version.",
    `\n=== CANONICAL VERSION (${canonicalBranch}) ===\n`, canonicalContent,
    `\n=== BRANCH VERSION (${entry.branchName}) ===\n`, branchContent,
].join("");
```

Last resort for cases where the foundational assumptions have changed since the agent started working.

### Self-Learning Merge Patterns

**File**: `src/merge/resolver.ts:405-554`

The system records every merge outcome to mulch and queries past patterns before attempting merges:

```typescript
// Record: fire-and-forget, never blocks the merge
mulchClient.record("architecture", {
    type: "pattern",
    description: `Merge conflict resolved at tier auto-resolve. Branch: overstory/auth-builder/task-123. Conflicting files: src/auth.ts.`,
    tags: ["merge-conflict"],
});

// Query: skip tiers that repeatedly fail for the same files
export function buildConflictHistory(patterns, entryFiles): ConflictHistory {
    // If a tier has failed >= 2 times for overlapping files and never succeeded, skip it
    for (const [tier, counts] of tierCounts) {
        if (counts.failures >= 2 && counts.successes === 0) {
            skipTiers.push(tier);
        }
    }
    return { skipTiers, pastResolutions, predictedConflictFiles };
}
```

This creates a feedback loop: attempt merge → record outcome to mulch → future merges query history → skip tiers known to fail for similar files → attempt merge with better strategy. The system bootstraps itself — it starts with no knowledge and gets smarter with each conflict.

## Watchdog: Health Monitoring & Recovery

### The ZFC Principle

**File**: `src/watchdog/health.ts`

ZFC (Zero Failure Crash): Observable state is the source of truth, not recorded state.

Signal priority (highest to lowest):
1. **tmux session liveness** — Is the tmux session actually running? (kernel truth)
2. **Process liveness (pid)** — Is the Claude Code process still alive? (kernel truth)
3. **Recorded state** — What does `sessions.db` claim? (maybe stale)

When signals conflict, always trust what you can observe:

| tmux | pid | DB says | Decision |
|------|-----|---------|----------|
| dead | - | "working" | Zombie immediately. DB is stale. |
| alive | - | "zombie" | Investigate. Don't auto-kill — something external marked it zombie but it recovered. |
| alive | dead | "working" | Zombie. Shell survived but agent process exited. |
| alive | alive | - | Check time-based thresholds. |

The rationale: `sessions.db` is updated asynchronously by hooks and can become stale if the agent crashes between hook invocations. tmux and the OS process table are always up-to-date because they reflect real kernel state.

### Health Evaluation State Machine

```typescript
// src/watchdog/health.ts:41-47
const STATE_ORDER: Record<AgentState, number> = {
    booting: 0,
    working: 1,
    completed: 2,
    stalled: 3,
    zombie: 4,
};

// State transitions are strictly forward-only
export function transitionState(currentState: AgentState, check: HealthCheck): AgentState {
    if (check.action === "investigate") return currentState;  // hold on conflicting signals
    const currentOrder = STATE_ORDER[currentState];
    const checkOrder = STATE_ORDER[check.state];
    if (checkOrder > currentOrder) return check.state;  // advance
    return currentState;                                  // hold
}
```

A zombie can never become working again. This prevents state flapping. The one exception: when `action === "investigate"` (ZFC signals conflict), state is frozen until a human or higher-tier agent reviews.

Persistent capabilities (coordinator, monitor) are exempt from time-based stale/zombie detection — they're expected to sit idle for long periods waiting for mail.

### Tiered Escalation

**Tier 0 — Mechanical daemon** (`src/watchdog/daemon.ts`): Runs on a fixed interval (default: 30 seconds). No AI, no cost. Uses implicit exponential backoff via time-bucket division:

```typescript
const expectedLevel = Math.min(
    Math.floor(stalledMs / nudgeIntervalMs),
    MAX_ESCALATION_LEVEL,
);
```

- Level 0: Log a warning
- Level 1: Send a tmux nudge ("are you alive?")
- Level 2: Escalate to Tier 1
- Level 3: Kill the session

**Tier 1 — AI-assisted triage** (`src/watchdog/triage.ts`): When Tier 0 escalates, `claude --print` analyzes recent logs and classifies the situation as recoverable, fatal, or long-running. Returns "retry," "terminate," or "extend." Falls back to "extend" if Claude is unavailable (safe default). On Max plan, this is free.

**Tier 2 — Monitor agent** (`src/commands/monitor.ts`): A persistent Claude Code session that correlates failures across the fleet and implements higher-level recovery strategies.

### Process Tree Cleanup

**File**: `src/worktree/tmux.ts:233-325`

When killing an agent, overstory walks the process tree depth-first and kills children before parents:

```typescript
export async function getDescendantPids(pid: number): Promise<number[]> {
    const childPids = /* pgrep -P pid */;
    const allDescendants: number[] = [];
    for (const childPid of childPids) {
        const grandchildren = await getDescendantPids(childPid);
        allDescendants.push(...grandchildren);  // grandchildren first
    }
    allDescendants.push(...childPids);  // then children
    return allDescendants;  // deepest-first order
}
```

Then the three-phase cleanup:

1. **SIGTERM** all descendants (deepest-first), then root
2. **Wait** grace period (2 seconds) for processes to clean up
3. **SIGKILL** any survivors (same order)

This prevents orphaned `bun test`, `biome`, or `git` processes from lingering after agent death. The depth-first ordering ensures children die before parents, preventing reparenting to init (PID 1).

## Guard Rules

**File**: `src/agents/guard-rules.ts`, `src/agents/hooks-deployer.ts`

Agents are mechanically constrained by their capability:

- **Read-only agents** (scout, reviewer, coordinator): No Write, Edit, or destructive Bash. Can only search, read, and send mail.
- **Builders**: Can modify files, but only those in their `FILE_SCOPE`. Cannot spawn sub-agents. Must pass quality gates before reporting done.
- **Leads**: Can spawn sub-agents but cannot exceed their configured `maxAgents` ceiling.

Guards are enforced through Claude Code's hook system (`PreToolUse` hooks that reject unauthorized tool calls) — they're not just instructions, they're mechanical blocks. The hook runs `ov` code that checks the tool name and arguments against the agent's allowed list. If it fails, the tool call is rejected with a clear error message before execution even begins.

## Runtime Abstraction

**File**: `src/runtimes/types.ts`, `src/runtimes/registry.ts`

Overstory is not locked to Claude Code. The `AgentRuntime` interface defines the contract that any AI coding agent must implement:

```
buildSpawnCommand()     → How to launch the agent in tmux

Current adapters: **Claude Code**, **Pi**, **Codex**, **Copilot**, **Gemini CLI**, **OpenCode**. Set the default in config or override per-agent with `ov sling --runtime <name>`.

## Algorithmically Interesting Code

### Hand-rolled YAML Parser

**File**: `src/config.ts:118-342`

Overstory has zero external YAML dependencies. Instead, it includes a stack-based, single-pass O(n) parser with:
- Indentation tracking via an explicit stack
- Quote-aware comment handling (doesn't strip `#` inside strings)
- Array/object ambiguity resolution
- Inline array parsing (`[a, b, c]`)

This eliminates a runtime dependency and handles overstory's config format with ~200 lines of code.

### Session Insight Analyzer

**File**: `src/insights/analyzer.ts`

Heuristic classification engine that runs at session end:
- **Path-based domain inference**: Maps file paths to expertise domains (e.g., `src/auth/` → `auth`)
- **Workflow categorization**: Classifies sessions by tool usage percentages (e.g., >60% Read = exploration session)
- **Hot-file detection**: Files edited >=3 times in one session are flagged
- **Error-tool correlation**: Pairs errors with the tool calls that produced them

Results are automatically recorded to mulch so future agents benefit from past sessions' learnings.

### Tool Arg Filtering

**File**: `src/events/tool-filter.ts`

Polymorphic per-tool strategy that reduces ~20KB tool payloads to ~200 bytes for observability storage. Each tool type has a custom filter extracting only what matters — file paths for Read/Write, patterns for Grep, command for Bash, etc.

## The Ecosystem

Overstory is part of [os-eco](https://github.com/jayminwest/os-eco), an integrated toolkit where each tool handles one concern:

```
overstory (orchestrates agents)
  ├── mulch    (stores expertise agents record and retrieve)
  ├── seeds    (tracks issues agents work on)
  └── canopy   (manages prompts agents use)
```

### Mulch — Structured Expertise

Agents start every session from zero. The pattern your agent discovered yesterday is forgotten today. Mulch fixes this by giving agents a persistent memory layer:

- **Record**: Agents call `ml record <domain> --type <type> "description"` to write learnings
- **Retrieve**: At session start, `ml prime` outputs accumulated expertise in AI-optimized format
- **Search**: `ml search` and `ml query` find relevant expertise mid-session

Key design: passive layer (contains no LLM), git-native (`.mulch/expertise/<domain>.jsonl`), multi-agent safe (advisory locks + atomic writes + `merge=union`).

Overstory integrates mulch at three points:
1. `ov prime` calls `ml prime` to inject expertise at session start
2. `ov sling` calls `ml prime --files <scope>` so spawned agents get file-relevant expertise
3. The merge resolver queries/records conflict patterns for self-learning

Mulch is the only ecosystem tool imported as a **programmatic API** (via `src/mulch/client.ts`), not just a CLI subprocess.

### Seeds — Issue Tracking

Git-native issue tracking. JSONL storage, dependency graphs, templates, zero external dependencies. Overstory uses seeds as the work queue: agents claim issues, report progress, and close completed work.

### Canopy — Prompt Management

Version-controlled prompt composition with inheritance, pinning, and schema validation. Bootstrapped at `ov init` time.

All four tools share the same design principles: git-native storage (JSONL/YAML, no external database), zero runtime dependencies (single Bun binary), multi-agent safe (file locking, WAL mode), and CLI-first (`--json` output on every command).

## Design Philosophy

### Cost awareness is a first-class concern

Every spawned agent is a full AI session costing real money (or Max plan quota). The hierarchy exists to multiply effectiveness per dollar — a coordinator spawning 3 leads that each spawn 4 builders yields 12 effective workers from 16 sessions, with clear scope preventing redundant work.

### Execution over deliberation

Every agent definition embeds the "propulsion principle": receive the objective, execute immediately. Don't ask for confirmation, don't propose a plan and wait. Start working within your first tool call. Failures can be escalated via mail; delays are worse than recoverable mistakes.

### Mechanical enforcement over instruction compliance

Guard rules are not suggestions — they're hooks that reject unauthorized tool calls before execution. File scope is enforced by checking paths against the agent's allowed list. Hierarchy depth is validated at spawn time. Trust the mechanical constraint, not the model's willingness to follow instructions.

### The honest case against swarms

Overstory ships with [STEELMAN.md](STEELMAN.md) — a document presenting the strongest arguments against multi-agent orchestration. Compounding error rates, cost amplification, debugging complexity, context fragmentation, premature decomposition. These are real risks, not edge cases.

Agent swarms have narrow but genuine use cases: truly independent tasks, embarrassingly parallel work, large-scale exploration, time-critical sprints. For most day-to-day engineering — deeply interconnected, requiring coherent reasoning — a single focused agent is often better. Overstory is a power tool. Know when to use it and when not to.

## Comparison: Overstory vs Claude Code Agent Teams

Claude Code now ships with [built-in agent teams](https://code.claude.com/docs/en/agent-teams) (experimental, behind a feature flag). Both systems solve the same fundamental problem — parallelizing AI coding work — but at very different levels of sophistication.

### Shared Foundations

Both systems converge on the same core ideas:

| Concept | Claude Code Agent Teams | Overstory |
|---------|------------------------|-----------|
| Lead + workers | One session is team lead, spawns teammates | Orchestrator → Coordinator → Lead → Workers |
| Shared task list | `~/.claude/tasks/{team-name}/` with file locking | SQLite `sessions.db` + issue tracker (seeds/beads) |
| Inter-agent messaging | Built-in mailbox (`SendMessage` tool) | Custom SQLite mail (`mail.db`) with typed protocol |
| tmux integration | Split-pane display mode | All agents run in tmux sessions |
| Graceful shutdown | Shutdown request/response protocol | Process tree cleanup (SIGTERM → grace → SIGKILL) |

### Key Differences

#### Hierarchy Depth

**Claude Code Teams**: Flat. One lead + N teammates. No nested teams — "teammates cannot spawn their own teams." The lead is fixed for the team's lifetime.

**Overstory**: 3+ levels deep. Orchestrator → Coordinator → Lead → Workers (Scout, Builder, Reviewer, Merger). Leads spawn sub-workers autonomously. Configurable `maxDepth` (default 2). This is the biggest architectural difference — overstory models a military chain of command, not a flat peer group.

#### File Isolation

**Claude Code Teams**: Teammates share the same working directory by default. The docs explicitly warn: "two teammates editing the same file leads to overwrites." Git worktrees are mentioned only as a manual technique in a separate docs page.

**Overstory**: Every agent gets its own git worktree automatically via `ov sling`. File isolation is **mandatory**, not optional. On top of that, `FILE_SCOPE` enforcement via `PreToolUse` hooks means a builder can only edit the specific files it was assigned — even within its own worktree.

#### Merge Strategy

**Claude Code Teams**: No merge system. If agents work in separate worktrees, merging is manual. The docs don't address conflict resolution.

**Overstory**: 4-tier automated merge resolution (clean → auto-resolve → AI-resolve → re-imagine) with a self-learning feedback loop via mulch conflict patterns. The merge resolver records outcomes and skips tiers known to fail for similar files. This entire subsystem has no equivalent in Claude Code teams.

#### Capability Enforcement

**Claude Code Teams**: All teammates inherit the lead's permission settings. No capability-based restrictions — every teammate can read, write, edit, and execute equally. You can change individual permissions after spawn, but there's no concept of "this agent is read-only by design."

**Overstory**: Mechanical enforcement via `PreToolUse` hooks deployed per-agent based on capability. Scouts can't write files. Builders can only write to scoped files. Reviewers are read-only. These aren't instructions the model might ignore — they're tool-call rejections that fire before execution.

#### Health Monitoring

**Claude Code Teams**: Manual. The docs advise: "check in on teammates' progress, redirect approaches that aren't working." Teammates may "stop after encountering errors instead of recovering."

**Overstory**: 3-tier automated watchdog:
- **Tier 0**: Mechanical daemon checking tmux/pid liveness every 30 seconds (no AI, no cost)
- **Tier 1**: AI triage via `claude --print` classifying failures as recoverable/fatal/long-running
- **Tier 2**: Persistent monitor agent correlating failures across the fleet

The ZFC principle (trust kernel state over database state) and forward-only state machine prevent flapping and ensure reliable failure detection.

#### Persistent Learning

**Claude Code Teams**: Teammates load `CLAUDE.md` and project context at spawn. No persistent learning across sessions.

**Overstory**: Mulch integration at three points:
1. Expertise injection at spawn (`ml prime --files <scope>`)
2. Session-end insight recording (conventions, patterns, failures)
3. Self-learning merge conflict patterns

Future agents benefit from past agents' discoveries.

#### Runtime Support

**Claude Code Teams**: Claude Code only. It's a native Claude Code feature.

**Overstory**: Runtime-agnostic via `AgentRuntime` interface. Adapters for Claude Code, Pi, Codex, Copilot, Gemini CLI, OpenCode. You can mix runtimes in the same swarm — e.g., Claude Code leads with Pi builders.

### When to Use Which

**Claude Code Agent Teams** when:
- You want zero setup — just enable the feature flag
- Your task is research, review, or parallel exploration (3-5 teammates)
- Teammates don't need strict file isolation
- Flat hierarchy is sufficient (1 lead + N workers)
- You don't need automated merge resolution
- You're staying within Claude Code

**Overstory** when:
- You need strict file isolation across many concurrent agents
- You need hierarchical decomposition (coordinator → leads → workers)
- You want automated merge with 4-tier conflict resolution
- You need capability enforcement (read-only scouts, file-scoped builders)
- You want health monitoring with automatic recovery
- You want persistent expertise that survives across sessions
- You need to orchestrate across multiple AI runtimes

### Complementary Use

The two systems aren't mutually exclusive. You could use Claude Code's native agent teams for quick parallel research within a single session, and bring in Overstory when you need the full orchestration machinery — deep hierarchy, mechanical enforcement, automated merging, self-learning, and multi-runtime support.

Claude Code Agent Teams is the **80% solution** — it handles the common case with zero infrastructure. Overstory is the **power tool** for the remaining 20% where the common case isn't enough.

## Summary: The Core Loop

```
validate → isolate (worktree) → instruct (overlay) → guard (hooks)
    → launch (tmux) → poll (readiness) → kick (beacon)
    → monitor (watchdog) → cleanup (process tree)
```

Everything else in the codebase — mail, merge, dashboard, costs, traces, replay — is infrastructure supporting this cycle.
